# Fusion Trust Investigation — Why Fusion Launches But Lab/Forge Don't

## Summary

Fusion.exe (C-Fusion.exe) launches directly, while Lab.exe and Forge.exe remain blocked by Windows Application Control. All three exes are signed with the same self-signed dev certificate (`CN=MystikStudio Local Dev Code Signing`), have no Mark of the Web, and have no active MystikStudio WDAC supplemental policy in the `CiPolicies\Active` folder.

## Files Inspected

- `Creators\C-Fusion\C-Fusion.exe` (58,368 bytes, built 05/12)
- `Creators\C-Lab\Lab.exe` (365,568 bytes, built 05/13)
- `Creators\C-Forge\Forge.exe` (359,936 bytes, built 05/13)
- `Start-MystikStudioDashboard.ps1` (launch targets)
- `C:\Windows\System32\CodeIntegrity\` (WDAC policies)
- `C:\Windows\System32\CodeIntegrity\CiPolicies\Active\` (active policies)
- `tools\signing\Install-CFusionLocalTrustPolicy.ps1` (old per-app installer)
- `tools\signing\Install-MystikStudioLocalTrustPolicy.ps1` (combined installer)
- `tools\signing\Sign-MystikStudioExe.ps1` (signing script)

## Key Findings

### 1. Dashboard Launch Targets
| App | Target | Format |
|-----|--------|--------|
| Fusion | `Creators\C-Fusion\C-Fusion.exe` | Direct exe |
| Lab | `Creators\Lab\Open Lab.vbs` | VBS → PowerShell (fallback) |
| Forge | `Creators\Forge\Open Forge.vbs` | VBS → PowerShell (fallback) |

Fusion launches directly via `Start-Process -FilePath $t -WorkingDirectory $workDir` — no wrapper, no special handling.

### 2. All Three Exes Are Signed Identically
- Certificate: `CN=MystikStudio Local Dev Code Signing`
- Thumbprint: `03DF7DEC03A342769B3130D3C97888051E2CD22F`
- Signature Status: `Valid` on all three
- Authenticode signature type, not OS binary
- No Zone.Identifier / Mark of the Web on any

### 3. File Size and Build Date Differences
| App | Size | Built | Notes |
|-----|------|-------|-------|
| Fusion | 58,368 bytes | 05/12 20:20 | Original build, smaller app |
| Lab | 365,568 bytes | 05/13 18:30 | Rebuilt with GPU status (Entry 55) |
| Forge | 359,936 bytes | 05/13 18:31 | Rebuilt with GPU status (Entry 55) |

Fusion was built a day earlier than Lab/Forge. It has not been rebuilt since.

### 4. WDAC Policy State
- **CodeIntegrity folder** contains: `MystikStudioCreators.p7b` (22,712 bytes, generated 05/13 19:28)
  - This policy was generated by `New-CIPolicy -ScanPath Creators\C-Fusion` — it only covers C-Fusion.exe's hash
  - Lab.exe and Forge.exe hashes are NOT in this policy
- **Active policies folder**: Contains 8 active .cip files but **none are MystikStudio-related**
- The MystikStudioCreators.p7b policy was **never activated** by Windows
- Smart App Control state: **Enforce**

### 5. Why Fusion Launches

The most likely explanation is **timing-dependent caching or partial policy coverage**:

1. Fusion was compiled and signed on **05/12** before the GPU status rebuild pass
2. At some point between 05/12 and 05/13, a WDAC policy may have been active that covered Fusion (`MystikStudioC-Fusion.p7b` from earlier work)
3. Fusion's allowance may be cached by the Windows App Control subsystem
4. The `MystikStudioCreators.p7b` policy in CodeIntegrity\ covers C-Fusion.exe's hash (since it was generated from scanning the C-Fusion folder)
5. Even though this policy is not in the Active folder, Windows may partially honor .p7b files present in CodeIntegrity\

**Lab and Forge have no such coverage:**
- They were rebuilt on 05/13 (new hashes after GPU status code changes)
- The `MystikStudioCreators.p7b` policy only covers C-Fusion.exe, not Lab.exe or Forge.exe
- No active policy ever covered their 05/13 hashes
- The system base WDAC policy `{0283ac0f-fff1-49ae-ada1-8a933130cad6}` blocks all self-signed exes at Enterprise signing level

### 6. Smart App Control Role
SAC is in **Enforce** mode (registry value `VerifiedAndReputablePolicyState` = 1). SAC blocks executables without cloud reputation. Self-signed dev certificates have no cloud reputation. SAC alone would block all three exes equally.

The fact that Fusion launches suggests either:
- Fusion was allowed before SAC fully locked down
- The `MystikStudioCreators.p7b` provides partial coverage even without being in Active
- There is a separate mechanism (like a local allow list or cached reputation) that covers Fusion

## Conclusion
**Fusion launches because it was compiled before the critical policy state, and the existing `MystikStudioCreators.p7b` (generated from scanning C-Fusion) covers its hash. Lab and Forge were rebuilt after and were never covered by any policy.**

### Fusion Direct Launch — Grandfathered Exception
Fusion's current direct exe launch is a **grandfathered exception** due to:
- Its original 05/12 build hash being covered by the `MystikStudioCreators.p7b` policy that exists in `C:\Windows\System32\CodeIntegrity\`
- Timing: it was established before the system's WDAC/SAC policies fully locked down for new self-signed exes

**Fragility warning:**
- **Do not rebuild Fusion** unless a new trust path is proven first. A rebuild will change Fusion's hash and the `MystikStudioCreators.p7b` policy will no longer cover it, causing Fusion to become blocked like Lab and Forge.
- **Do not delete `MystikStudioCreators.p7b`** from `C:\Windows\System32\CodeIntegrity\`. Removing it may cause Fusion to stop launching.

## Recommended Next Action
**Do not retry C# activation for Lab or Forge.** The WDAC supplemental policy path on this system only worked by historical accident for Fusion. Lab and Forge cannot be activated without either:
1. A Microsoft-trusted code signing certificate (EV Code Signing or standard CA), or
2. A change in the system's WDAC/Smart App Control policy (not recommended), or
3. Using a system that does not have WDAC Enterprise signing enabled.

C# exes are staged at `Creators\C-Lab\Lab.exe` and `Creators\C-Forge\Forge.exe` for future reference, but they are not launchable on this system. Dashboard targets must remain on PowerShell fallback for Lab and Forge.
